Telegram Video Archive Bot

A private Telegram bot that copies forwarded videos into my own archive channel, indexes them in SQLite, and lets me sort everything by category without re-uploading gigabytes of files.

Junye Zhou · solo build Python · aiogram 3.x SQLite · private channel storage Cursor AI wrote ~100% of the code

What it does

I forward a video (or a whole album) to my bot in a private chat. The bot copies the message into a dedicated archive channel I control, stores metadata in a local database, and asks which category the upload belongs in. Later I can browse by category, search captions, or open the original archived message in Telegram with one tap.

The point is not “another downloader.” Telegram already hosts the files. What I wanted was a personal index that survives when a source channel disappears, plus a workflow that feels as low-friction as forwarding a message.

Why I built it

I use Telegram for a lot of video content — tutorials, clips, references. Saved Messages helps, but it turns into a pile fast. Worse, when a channel gets deleted, restricted, or the account vanishes, the forward you kept might still play… until it does not. I wanted something closer to a real archive: my copy, my channel, my labels.

The design goals were simple:

  • Save by forwarding, not by manual download/upload
  • Keep the heavy files in Telegram (use file_id + channel copies)
  • Organize with categories, optional tags, and album-style collections
  • Stay usable when I dump 20 videos in a row (batch categorization)

Technical stack

Python 3 aiogram 3.x SQLite (aiosqlite) Telegram Bot API

The bot runs as a long-polling Python process on my machine (or a VPS when I move it). aiogram handles updates and inline keyboards. SQLite stores one row per archived video: file_id, archive message id, caption, source chat info, category, collection link, and a bulk_upload_id for batch forwards.

Storage strategy: the “real” video lives in a private Telegram channel the bot administers. The database is only an index — like a library catalogue, not the warehouse. That keeps the bot lightweight and avoids burning disk space on multi-gigabyte files.

How it was built: I did not hand-write this codebase line by line. Cursor AI wrote nearly 100% of the code — project setup, handlers, database migrations, bug fixes, and UI tweaks. My role was to describe what I wanted in plain language, test the bot in Telegram, and send follow-up prompts when something broke or felt annoying. I still own the product decisions (private channel storage, categories, batch flows), but the implementation is almost entirely AI-generated inside Cursor.

Other Telegram bots I used

Besides writing my own bot in Python, setup relied on a few official and third-party Telegram bots. None of them store my videos — they only helped with credentials and IDs.

@BotFather

What it is: Telegram’s official bot for creating and managing bots.

How I used it:

  • /newbot — create the bot and get the BOT_TOKEN for .env
  • /mybots — open bot settings, revoke/regenerate the token if it leaks
  • Add the bot as an admin of my private archive channel (must be allowed to post)

BotFather only manages the bot account. It does not give you channel IDs or archive videos for you.

@raw_data_bot (Raw Data Bot)

What it is: A utility bot that replies with the raw JSON Telegram sends about a message or chat — useful for debugging and for copying numeric IDs.

How I used it: After adding it to my private archive channel, it printed the channel’s chat id in the -100… format. That value goes in ARCHIVE_CHANNEL_ID in .env. This was more reliable for channels than forwarding a message to a generic “get id” bot.

@userinfobot

What it is: Shows your Telegram user id and basic info when you forward a message or start a chat.

How I used it: To find my own OWNER_USER_ID (optional in .env so only I can use the bot). Important distinction: a user id is a positive number like 5856941941; a channel id is negative and usually starts with -100. I initially confused the two — userinfobot is for people, not channels.

@getidsbot (optional)

Same job as raw_data_bot for some setups: add it to a group or channel, and it reports chat ids. I mention it because tutorials often recommend it; I ended up preferring raw_data_bot for the archive channel.

How to activate and use the bot

These steps match how the project is meant to run on Windows (local machine). For 24/7 uptime, run the same code on a PC that stays on, or move it to a VPS later.

1 — Create the bot (Telegram)

  1. Open @BotFather → /newbot → copy the token.
  2. Create a private channel (your archive storage).
  3. Add your bot to that channel as Administrator with permission to post messages.
  4. Add @raw_data_bot to the channel briefly; copy the channel id (-100…).
  5. Optional: message @userinfobot to get your user id for OWNER_USER_ID.

2 — Configure the project

In the project folder, copy .env.example to .env and set:

BOT_TOKEN=your_token_from_botfather
ARCHIVE_CHANNEL_ID=-100xxxxxxxxxx
OWNER_USER_ID=your_numeric_user_id
SQLITE_PATH=data/bot.sqlite3

Never commit .env or paste the token in chat. If it leaks, revoke it in BotFather.

3 — Install and start

cd videodownloadbot
py -m venv .venv
.\.venv\Scripts\pip install -r requirements.txt
# Or double-click start_bot.bat on Windows

Leave the terminal window open while the bot runs. Use stop_bot.bat or close the window to stop it. If you see “conflict” errors, only one instance should be running — run taskkill /IM python.exe /F once, then start again.

4 — Daily usage

  • Save a video: Open your bot in DM → forward (or send) a video. Wait for “Saved as #id” → tap a category on the inline keyboard.
  • Batch forwards: Forward many videos one after another; after a short pause you get one picker for the whole batch.
  • Album / media group: Forward an album; the bot saves a collection and asks for a category once.
  • Skip → Uncategorized: Keep the save, label as uncategorized.
  • Cancel · Don't save: Remove that upload from the archive and database.

5 — Commands

/start, /help     — intro and command list
/videos            — browse by category (paginated)
/search <text>    — search captions, tags, categories
/move <id>        — change category (e.g. /move 12 zw)
/tag <id> …       — set category + extra tags
/collection <id>  — open an album collection
/delete <id>      — remove from archive + database
/stats             — count and storage summary

Protected or expired forwards may fail to copy; download the file to your phone and send it as a file to the bot instead of forwarding.

Key features

Forward-to-save

Send or forward any supported video (including many video documents). The bot prefers copy_message into the archive channel — server-side, no local download — which also works for large files. If copy fails, it falls back to download + re-upload with FSInputFile (Windows paths taught me that lesson quickly).

Batch grouping & collections

Forwards sent one after another are debounced into a bulk batch; I get one category picker for the whole set. Albums (media_group_id) become a collection: each item is saved, linked in SQLite, and a summary post is added to the archive channel with links to every item.

Categorization UI

After a save, an inline keyboard lists existing categories (sorted by video count), plus “New category,” “Skip → Uncategorized,” and “Cancel · Don’t save.” Cancel on a fresh forward removes that upload from the archive and database — useful when I forwarded something by mistake. Assigning a category updates every video in the batch via bulk_upload_id, so late-finishing saves still land in the right folder.

Browse & search

/videos opens a category menu. Inside a category I get a paginated list (links to archive messages), a 10-video preview album per page, and compact page controls under the grid so I do not scroll back to the top. /search runs a simple SQL LIKE over captions, tags, source titles, and categories. /move re-categorizes one or many ids without re-archiving.

Tags & maintenance commands

/tag sets a primary category plus extra tags on one video. /delete, /stats, and guarded wipe commands exist for housekeeping. Owner-only checks mean random users cannot touch the archive.

Code highlights

Telegram bots do not receive “the file on disk.” They receive a file_id — an opaque handle the API uses to re-send the same media later. Storing file_id plus the archive channel’s message_id is how I can rebuild links and preview albums months later.

Handling incoming video

@dp.message(F.video | F.document)
async def handle_video(m: Message) -> None:
    extracted = _extract_video(m)
    if not extracted:
        return
    file_id, file_unique_id, file_size, caption, kind = extracted

    copied = await bot.copy_message(
        chat_id=cfg.archive_channel_id,
        from_chat_id=m.chat.id,
        message_id=m.message_id,
    )

    video_id = await db.insert_video(
        file_id=file_id,
        file_unique_id=file_unique_id,
        archive_chat_id=cfg.archive_channel_id,
        archive_message_id=copied.message_id,
        bulk_upload_id=int(bulk["bulk_id"]) if bulk else None,
        ...
    )

copy_message creates an independent message in my channel, so the archive does not depend on the original chat staying reachable. The insert gives me a stable integer #id for commands like /move 42 zw.

Album / media-group batching

if m.media_group_id:
    key = (m.chat.id, str(m.media_group_id))
    grp = pending_groups.setdefault(key, _PendingGroup(...))
    grp.messages.append(m)
    grp.finalize_task = asyncio.create_task(
        finalize_media_group(key)
    )
    return

Telegram delivers album items as separate updates sharing a media_group_id. I buffer them, wait for the burst to finish, then save all items and create one collection row. Without the delay, I would save incomplete albums.

Bulk category assignment

elif target["t"] == "b":
    bulk = pending_bulk_by_id.get(int(target["id"]))
    if not bulk:
        return
    await db.set_primary_category_for_bulk_upload(
        int(bulk["bulk_id"]), category
    )

Early versions tracked video ids only in memory — if I categorized before the last save finished, a few videos stayed uncategorized. Tagging each row with bulk_upload_id at insert time and updating by that id fixed the race for good.

Challenges & learnings

  • Async timing. Multiple forwards mean overlapping handlers. I track in_flight saves and debounce the category picker so it appears once the batch actually settles.
  • Telegram quirks. Protected content cannot be copied. Large files may force a download path. Callback data has a tight length limit, so category names are normalized and truncated.
  • UX on mobile. Pagination for /videos needed preview grids and duplicate page controls under each grid — otherwise I was scrolling forever to change pages.
  • Keeping the index honest. Cancel-on-upload deletes both the DB row and the channel message so “mistake forwards” do not clutter the archive.

What I would add next

  • Deploy on a small VPS with systemd so it runs 24/7 without my laptop
  • Optional cloud backup (S3 / Drive) for metadata exports
  • Simple web dashboard to browse categories outside Telegram
  • Auto-tagging from captions (keyword rules or a small LLM pass)

For now it does what I needed: forward, label, find. That is enough to call it a successful side project — and a good excuse to learn asyncio properly.
